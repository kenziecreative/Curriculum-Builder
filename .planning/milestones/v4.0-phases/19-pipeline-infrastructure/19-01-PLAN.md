Add a "Registry Write" section to each stage command, positioned immediately after the file-write step (where the command writes its stage output to disk) and before the completion summary or handoff message. Also add registry write logic to approve.md for both gate types.

**Pattern for each stage command (add after existing file writes, before completion summary):**

```markdown
## Registry Write

After writing stage files, update `workspace/{project}/curriculum-registry.json`:

1. Read the existing registry file. If it does not exist (first stage to run), create it with the `meta` section populated from the project name and current timestamp, schema_version "1.0".
2. Write/overwrite the `{section}` key with data extracted from the files just written.
3. Set `{section}.last_updated` to current ISO datetime.
4. Set `{section}.stage_source` to {N}.
5. Write the registry as formatted JSON (2-space indent) so an SME opening the file can read it.

Do this silently -- no announcement to the user.
```

**Per-command specifics (what each writes):**

- **intake.md** (Stage 1): Writes `learner_profile` section. Extract from the project-brief.md just written: target_audience description, expertise_level, self_direction_level, skill_type, cultural_orientation, transfer_context, contact_hours, modality, success_criteria. Also creates the `meta` section since intake is always the first stage. Add the registry write instruction in the "Write Project Brief" section (around the area where it writes project-brief.md to disk), after the file write but before the "Type `/clear` now" handoff message.

- **outcomes.md** (Stage 2): Writes `outcome_wording` section. Extract all program, module, and session outcomes with their IDs, statements, bloom_levels, and parent references from the learning-objectives.md just written. Add after the file write in the approval branch.

- **assessments.md** (Stage 3): Writes `assessment_criteria` section. Extract assessment IDs, types (map formative->check-in, summative->final for the registry), titles, mapped outcome IDs, and criteria summaries. Add after file write in the approval branch.

- **modules.md** (Stage 4): Writes `time_allocations` section. Extract module IDs, names, sessions_planned counts, and hours_allocated from module specs. Add after file write in the approval branch.

- **sessions.md** (Stage 5): No new registry section -- sessions don't add a new category. But after File Verification passes, update `time_allocations.modules[].sessions_completed` with actual session counts per module. Add after the file verification step.

- **metaskills.md** (Stage 6): No new registry section. Registry is not modified by this stage. Skip.

- **transfer.md** (Stage 7): No new registry section. Registry is not modified by this stage. Skip.

- **marketing.md** (Stage 8): No new registry section. Registry is not modified by this stage. Skip.

- **approve.md**: Add registry update logic in BOTH gate types (post-assessment and final validation). When the user approves with revisions that changed outcome wording, assessment criteria, or module structure, the approve command must re-extract the changed data and update the corresponding registry section with a new `last_updated` timestamp. Add this after the "write revised files" step in approve.md's revision flow. The instruction should say: "If revisions changed content in a stage that has a registry section (outcomes, assessments, modules), re-read the revised files and update the corresponding registry section. Set last_updated to now."

**Important constraints:**
- Registry writes are SILENT -- never announce "Updating registry" to the user
- Use the same "Do this silently" pattern already established for STATE.md updates
- Reference the schema at `.claude/reference/schemas/curriculum-registry-schema.md` in each registry write instruction so the command knows the exact JSON structure
- For audit-mode workspaces: the same write hooks apply -- the registry builds incrementally as each stage finalizes
