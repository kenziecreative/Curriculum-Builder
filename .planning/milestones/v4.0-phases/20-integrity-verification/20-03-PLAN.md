<objective>
Add a cross-stage integration check to approve.md that traces every outcome ID, assessment link, and module reference across all 8 stages before the final approval gate, catching broken links and orphaned references before the package ships.

Purpose: Individual stage checks verify internal consistency, but no existing check verifies that references BETWEEN stages are valid. An outcome could be defined in Stage 2, referenced in Stage 4, but missing from Stage 5 sessions — and no current check catches this. The cross-stage integration check is the structural backbone verification that ensures the curriculum hangs together as a whole.

</objective>

Place the new section after the existing "For Final Validation gate only — run verify silently first" block. Add a parallel instruction:

```
**Also for Final Validation gate — run cross-stage integration check:**

Before showing any summary, trace these three link types across all stages. Read `workspace/{project}/curriculum-registry.json` as the canonical source. Then read the actual stage files to verify consistency. Store findings as `integration_findings`.
```

**The three traces:**

**Trace 1: Outcome IDs (bidirectional)**
- Forward: For every outcome ID in `outcome_wording.program_outcomes` and `outcome_wording.module_outcomes`, verify it appears in at least one assessment's `linked_outcomes` (Stage 3), at least one module's `learning_objectives` (Stage 4), and at least one session's content (Stage 5). An outcome that exists in the registry but is never assessed, never assigned to a module, or never taught in a session is an orphaned outcome.
- Backward: For every outcome ID referenced in assessments, modules, or sessions, verify it exists in the registry's `outcome_wording`. An ID that appears in a downstream stage but not in the registry is a broken reference.
- For stages not yet generated: mark as "Pending — Stage N not yet generated" (not broken).

**Trace 2: Assessment Links (bidirectional)**
- Forward: For every assessment in `assessment_criteria.assessments`, verify its `linked_outcomes` reference real outcome IDs AND that the assessment type (formative/summative) is appropriate for the stage it appears in.
- Backward: For every outcome ID, verify at least one assessment links to it. An outcome with no assessment is an uncovered outcome.
- For stages not yet generated: mark as pending.

**Trace 3: Module References (bidirectional)**
- Forward: For every module in `time_allocations.modules`, verify it has a corresponding module spec directory (Stage 4) and session directories (Stage 5). A module in the registry with no files is a phantom module.
- Backward: For every module directory that exists on disk, verify it has a corresponding entry in the registry. A directory with no registry entry is an unregistered module.
- Verify `prerequisite_modules` references in module specs point to real module IDs. A prerequisite that references a non-existent module is a broken dependency.

**Registry-file drift detection:**
After all three traces, compare the registry data against the actual stage files for any discrepancies (e.g., outcome statement in registry differs from outcome statement in 01-outcomes/ files). Report drift as a WARNING (not blocking), per user decision: "Registry-file drift is a warning, not a blocking failure — matches Phase 19 'registry wins' philosophy; user decides if it matters."

**Result classification:**
- Broken references (ID exists in one stage but not another where it should): BLOCKING
- Orphaned references (defined but never used downstream): BLOCKING
- Phantom/unregistered modules: BLOCKING
- Registry-file drift: WARNING
- Pending (stage not yet generated): INFORMATIONAL — not a failure

Store all findings in `integration_findings` with this structure:
```
{
  blocking: [{ type, description, source_stage, target_stage, id }],
  warnings: [{ type, description, details }],
  pending: [{ stage, reason }]
}
```
