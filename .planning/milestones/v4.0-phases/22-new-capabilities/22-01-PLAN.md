Purpose: After a curriculum has been delivered and lived with, SMEs need to refine it based on what actually happened — learner struggles, pacing issues, content gaps. This command provides the re-entry point without requiring a full rebuild.

  <name>Task 1: Write /curriculum:revise command — feedback capture and stage mapping</name>
  <files>.claude/plugins/curriculum/commands/revise.md</files>
  <action>
Create `.claude/plugins/curriculum/commands/revise.md` — the full command file. Follow the established command structure pattern from sessions.md and approve.md.

**Frontmatter:** `description: Revise a delivered curriculum based on post-delivery feedback — maps observations to affected stages, updates registry, and regenerates downstream files`

**Persona section:** Reference curriculum-voice.md. Tone: skilled colleague helping someone refine what they shipped. Never: pipeline stage numbers as user-facing labels, schema field names, ID terminology.

**Writing for Clarity block:** Identical to other commands — kernel sentences, active voice, no warm-up openers, precise language.

**Never-say list:** Same inline guardrail pattern as approve.md — reference curriculum-voice.md prohibited terms. Add context-specific note: never say "regeneration pipeline", "propagation engine", "downstream cascade" — use plain alternatives like "update", "revise the affected files", "make matching changes".

**Prerequisites section:**
1. Workspace detection (same pattern as sessions.md: find STATE.md, extract project name)
2. Directory scheme detection (same 00-08 vs 01-09 pattern)
3. Completion check: Read workspace STATE.md — ALL stages through Stage 8 must show `complete` status. If not all complete: "This command is for revising a curriculum that has been fully built. Run the remaining stages first, or use /curriculum:approve to make changes during the build."
4. Read curriculum-registry.json into context — needed for stage mapping and change propagation

**Step 1: Feedback Capture**
- Open with: "What did you learn after delivering this curriculum? Describe what changed, what surprised you, or what you'd do differently."
- Accept free-text description. No stage picker, no structured form. SMEs think in problems.
- Optional category tagging via AskUserQuestion after description: "Would any of these help me focus? Pick all that apply or skip."
  - Learner feedback (things learners said or struggled with)
  - Facilitator observations (what the instructor noticed during delivery)
  - Content updates (new information, outdated material)
  - Scope changes (add/remove/restructure)
  - Skip — just work from what I described

**Step 2: Gap Probing**
- After initial description, ask 2-3 targeted follow-up questions about areas the user did NOT mention. Questions are Claude's discretion based on what was described. Examples:
  - If user mentioned content but not pacing: "How did the timing feel? Any sessions that ran long or short?"
  - If user mentioned sessions but not transfer: "Did learners apply what they learned back on the job? Anything that didn't transfer?"
  - If user mentioned nothing about assessments: "How did the check-ins and final assessments land? Anything feel too easy or too hard?"
- Maximum 3 follow-up questions. Use AskUserQuestion with free-text input for each.

**Step 3: Categorize and Map**
- Categorize each feedback item as **fix** (something didn't work) or **evolution** (something worked but could be better)
- Map each item to affected stage(s) by reading workspace files and registry to identify where the relevant content lives
- For each affected stage, identify the specific files that need changes and what the changes would be
- Build an impact map showing: feedback item → affected stage(s) → specific files → proposed change → downstream effects

**Step 4: Show Impact Map and Confirm**
Present the impact map to the user. Format:

```
## What I'd change

### Fixes (things that didn't work)

**[Feedback summary]**
- Stage affected: [plain stage name, not number]
- Files to update: [filename list]
- What changes: [one-sentence description]
- Also updates: [downstream files that need matching changes]

### Improvements (things that could be better)
[Same format — these are optional; user can skip any]
```

Use AskUserQuestion:
- "Go ahead with all changes" — proceed to Step 5 for all items
- "Let me pick which ones" — present each item individually with yes/no
- "I want to change something" — return to feedback capture with additional input

**Step 5: Execute Changes**
For each approved change, in registry-first order:

1. **Update registry** — Load curriculum-registry-schema.md for exact JSON structure. Modify the affected registry section(s). Set `last_updated` to current ISO datetime. Write formatted JSON (2-space indent). Silent — no announcement.

2. **Identify downstream files** — Using registry parent references and outcome IDs, identify every file that reads from the changed registry section. Only include files whose content would actually change based on the registry update. Show the user: "These files need to be updated to match: [list]. Regenerate all?" Use AskUserQuestion: "Yes, regenerate all" / "Let me pick" / "Skip — I'll update them manually".

3. **Targeted regeneration** — For each file to regenerate:
   - Set the relevant stage status back to `in-progress` in workspace STATE.md
   - Regenerate to the stage's `_drafts/` directory using the same draft-then-audit pipeline the original stage command uses (same checks, same 3-attempt retry, same auto-fix boundary: vocabulary substitution, registry default fills, outcome drift correction)
   - On audit pass: promote from `_drafts/`, set stage status back to `complete`
   - On audit fail after 3 attempts: escalate to user with the same format as the original stage command's escalation

4. **Write revision log** — Append to `workspace/{project}/revision-log.md` (create if doesn't exist):
```markdown
## Revision: {date}

### Changes Applied

| What changed | Type | Stage | Files updated |
|-------------|------|-------|---------------|
| [feedback summary] | Fix / Improvement | [stage name] | [file list] |

### Downstream Updates
- [list of regenerated files with pass/fail status]
```

**Step 6: Completion and Loop**
After all changes are applied:
> Your revisions are complete. [N] files updated, [M] downstream files regenerated.
>
> Anything else to revise?

Use AskUserQuestion:
- "Yes, I have more feedback" — return to Step 1
- "No, I'm done" — show final summary and stop

**Important implementation notes:**
- This command is explicitly for POST-DELIVERY revision. The prerequisites enforce this by requiring all stages complete. It is NOT for mid-build corrections (that's approve.md's "I have concerns" path).
- Registry-first data flow: registry is updated BEFORE any file regeneration, consistent with Phase 19 "registry wins" principle.
- Targeted regeneration only: if outcome MO-2-1 changed, only sessions under Module 2 get regenerated, not all sessions. Use registry parent references to identify scope.
- The draft-then-audit pipeline is described by reference to the pattern, not re-specified. The executor should read the relevant stage command (sessions.md, metaskills.md, transfer.md, marketing.md) to understand the specific audit checks for that stage.
- Directory scheme detection must be included so paths resolve correctly for both legacy and new workspaces.
