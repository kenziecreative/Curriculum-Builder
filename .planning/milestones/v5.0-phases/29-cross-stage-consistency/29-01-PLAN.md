  <action>
Create `.claude/reference/consistency-check-reference.md` following the structural pattern of `alignment-check-reference.md`. The document must contain these sections:

**Section 1: Purpose and Scope**
- Consistency verification confirms that output across pipeline stages does not contradict itself — time allocations match, prerequisites are honored, and marketing claims trace to measured outcomes.
- When it runs: After alignment check passes (or after alignment check is skipped if no source material). Same 3-attempt retry with cumulative constraints pattern.
- What it checks against: `curriculum-registry.json` as the canonical source for all comparisons.
- When it is skipped: Each per-stage check is skipped if the upstream stage it checks against has not been generated yet. The final gate sweep runs all checks.

**Section 2: Per-Stage Consistency Checks**
Table format showing which checks run when each stage completes. Only stages that have cross-stage dependencies get checks:

| Stage | Checks |
|-------|--------|
| Sessions (Stage 5) | Time math vs module specs, prerequisite ordering, outcome ID coverage |
| Marketing (Stage 8) | Claim-to-assessment chain (claim -> outcome ID -> assessment exists) |
| Final Gate (approve) | Full sweep: all per-stage checks + outcome ID bidirectional trace + assessment link trace + module reference trace + registry-file drift |

Stages 2-4 and 6-7 do NOT get per-stage consistency checks — their cross-stage relationships are fully covered by the final gate sweep and by existing registry consistency checks. State this explicitly so Plan 02 implementors do not add unnecessary checks.

**Section 3: Session-to-Module Consistency (XSTAGE-01)**
Three check types:

3a. **Time Math** — For each module, sum the session durations (from session files in the module's session directory). Compare against the module spec's allocated hours (from `curriculum-registry.json` `time_allocations.modules`). On mismatch:
- BLOCKING
- Report format: table showing Module Name | Module Spec Hours | Session Total Hours | Difference
- Two fix paths in the report: "Run `/curriculum:modules` to update the module spec" OR "Run `/curriculum:sessions` to regenerate sessions within the budget"
- The SME picks the direction — the tool does not decide

3b. **Prerequisite Ordering** — For each session, check if it references outcomes from a module that comes later in the prerequisite chain. Read `prerequisite_modules` from module specs. If Session X in Module A references an outcome belonging to Module B, and Module A does not list Module B as a prerequisite, flag as a prerequisite ordering violation.
- BLOCKING
- Report: Session file path, the outcome ID referenced, the module it belongs to, and why the ordering is wrong
- Only ordering violations — NOT implicit knowledge gap detection

3c. **Outcome ID Coverage** — For each module, verify that every outcome ID listed in the module spec appears in at least one session within that module. An outcome assigned to a module but never taught in any session is a coverage gap.
- BLOCKING
- This is the per-stage version of the approve gate's existing orphaned outcome check, scoped to the sessions-modules relationship

**Section 4: Marketing Claim Tracing (XSTAGE-02)**
Every marketing claim links to an outcome ID (Phase 28 alignment check enforces this). This check verifies the next link in the chain: the linked outcome ID must have at least one assessment in `curriculum-registry.json` `assessment_criteria.assessments[].linked_outcomes`.

- If a marketing claim links to an outcome that has no assessment: BLOCKING
- Report: "Marketing promises '{claim text}' via outcome {ID}, but no assessment measures that outcome"
- Strictness is structural (outcome ID linkage), not semantic — marketing can use emotional/aspirational language

**Section 5: Final Gate Sweep**
The approve gate runs a comprehensive sweep that includes:
1. All per-stage checks from Sections 3-4 (re-run even if they passed during per-stage audit)
2. Outcome ID bidirectional trace (forward: every outcome used somewhere; backward: every reference points to real outcome)
3. Assessment link bidirectional trace (every outcome has at least one assessment; every assessment links to real outcomes)
4. Module reference trace (every module has files; every file has registry entry; prerequisites point to real modules)
5. Registry-file drift detection (WARNING, not blocking — registry is authoritative)

This consolidates the logic currently inline in approve/SKILL.md lines 123-159. The approve gate will load this reference instead of having its own inline logic.

Result classification (same as existing):
- Broken references: BLOCKING
- Orphaned outcomes: BLOCKING
- Phantom/unregistered modules: BLOCKING
- Registry-file drift: WARNING
- Pending (stage not generated): INFORMATIONAL

**Section 6: Contradiction Report Format (XSTAGE-04)**
Per user decision: table format for side-by-side contradictions. Reports grouped by contradiction type.

```
### Consistency Check Results

**Status:** PASS / FAIL ({N} issues found)

{For each contradiction type that has issues:}

**{Type}: {e.g., Time Mismatch / Prerequisite Violation / Unverifiable Marketing Claim / ...}**

| Source | Says | File |
|--------|------|------|
| {canonical source, e.g., "Module 2 spec"} | {value from source, e.g., "6 hours"} | {file path} |
| {conflicting source, e.g., "Sessions total"} | {conflicting value, e.g., "8 hours"} | {file path(s)} |

{Fix guidance for this type}
```

Plain language throughout. No internal codes. Grouped by contradiction type for batch-fixing.

**Section 7: Retry Constraint Format**
Same pattern as alignment-check-reference.md Section 7. Cumulative constraints. Specific per-issue instructions:
- Time mismatch: "Total session time for Module {N} is {X} hours but module spec allocates {Y} hours. Adjust session durations to fit within {Y} hours."
- Prerequisite violation: "Session {S} references outcome {ID} from Module {M2}, but Module {M1} does not list {M2} as a prerequisite. Either remove the reference or restructure the session to not depend on that outcome."
- Coverage gap: "Outcome {ID} is assigned to Module {N} but no session teaches it. Add coverage for this outcome in at least one session."

**Section 8: Integration Pattern**
- Sequence: Consistency check runs after alignment check (or after the check that would precede alignment if alignment is skipped). It is the final check in the draft audit sequence.
- Retry: Same 3-attempt with cumulative constraints. Issues from attempt 1 carry into attempt 2, etc.
- Auto-fix rule: Consistency issues are never auto-fixable — they require re-generation or upstream stage changes.
- Escalation: Same pattern as alignment — plain language, draft stays in `_drafts/`, specific issues listed.
- Trail write: Consistency Check subsection written only after check passes. Omitted if skipped or escalated.
