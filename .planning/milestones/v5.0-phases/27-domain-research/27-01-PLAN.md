**Structure (follow intake/SKILL.md pattern precisely):**

1. **Output Formatting** ‚Äî reference `curriculum:interactive-design` skill
2. **Title** ‚Äî `# /curriculum:research`
3. **Prerequisites** section:
   - Workspace check (same pattern as intake)
   - Directory scheme detection (same table as intake/outcomes)
   - Stage status check: Read the "Domain Research" row from STATE.md Stage Progress table. Handle `not-started` (proceed), `in-progress` (resume from last hypothesis), `complete` (respond with "already complete, run /curriculum:outcomes")
   - Stage 1 prerequisite check: Stage 1 must be `complete`
   - Source material skip check (RSRCH-06 integration point ‚Äî checked here but the actual routing happens in Plan 02's intake chaining. Include a guard here too): If `workspace/{project}/source-material/` already contains files, respond: "Your project already has source materials. Domain research is for builds starting without existing materials. Run `/curriculum:outcomes` to continue." Stop here.
   - Registry read: Load `curriculum-registry.json` for `learner_profile.data` ‚Äî extract `program_topic`, `target_audience`, `skill_type`, `transfer_context` to use as research context

4. **Persona** ‚Äî Read `.claude/reference/curriculum-voice.md`. "You are a research partner testing assumptions before the build begins. Your tone is curious, rigorous, and collaborative ‚Äî presenting evidence as a thinking partner, not a judge."

5. **Writing for Clarity** ‚Äî Same kernel sentences block as other stages.

6. **Hypothesis Gathering** section (RSRCH-01):
   - Opening message (natural extension of intake, not jarring): "Before we design this, one more thing. You know [domain from registry] ‚Äî I need to understand what you believe about it that should shape how people learn it."
   - Two question prompts via `AskUserQuestion` as free text (not multiple choice):
     a. "What do you believe about [domain] that should shape how people learn it? What do most people get wrong about learning this?"
     b. "Are there trends or shifts in [domain] right now that a curriculum should account for?"
   - After collecting responses, parse into distinct testable claims. Present back: "Here's what I'm hearing as claims to test:" followed by numbered list.
   - Use `AskUserQuestion` with options: "These look right", "I want to edit or add to these", "Start over"
   - Guardrails: If fewer than 3 claims parsed, ask: "Can you think of any other beliefs about how [domain] works or how people learn it?" If more than 7, suggest: "That's a lot of ground to cover. Can we consolidate any of these, or are they all distinct enough to research separately?"
   - Capture each claim as a testable statement (reframe opinions into testable form if needed)
   - Store claims internally for the research phase

7. **Research and Evidence Tagging** section (RSRCH-02, RSRCH-03):
   - For each hypothesis, use `tavily_search` (or `tavily_research` for broader queries) to find 2-4 sources
   - Apply evidence label AS EACH HYPOTHESIS IS RESEARCHED (not after all research ‚Äî per RSRCH-03):
     - **SUPPORTED**: Evidence broadly confirms the claim
     - **COMPLICATED**: Evidence partially confirms but adds important nuance or conditions
     - **CONTRADICTED**: Evidence suggests the opposite or a fundamentally different picture
     - **GAP**: Not enough evidence found to evaluate ‚Äî domain may be too niche or claim too specific
   - For each hypothesis, capture: the original claim, 2-4 source summaries with URLs, the evidence label, a brief synthesis (2-3 sentences)
   - Progress indicator: After each hypothesis is researched, show: "Researched {N} of {total}: [claim summary] ‚Äî {LABEL}" (use plain language labels from CONTEXT.md: "Supported by evidence", "It's more complicated than that", "Evidence suggests otherwise", "Not enough evidence found")
   - Do NOT present full findings yet ‚Äî just progress indicators

8. **SME Review Checkpoint** section (RSRCH-04):
   - Present all findings in a structured review format. For each hypothesis:
     ```
     ### {N}. {Original claim}
     **What you said:** {SME's original statement}
     **What I found:** {2-3 sentence synthesis}
     **Evidence:** {Label in plain language}
     **Key sources:** {Bulleted list of source titles with URLs}
     ```
   - For CONTRADICTED findings, add: "The evidence suggests otherwise. Does this change your thinking, or do you have context the research missed?"
   - After presenting all findings, use `AskUserQuestion` with free text: "Review each finding above. For any you want to adjust or add context to, tell me the number and what you'd like to change. If they all look right, say 'approved'."
   - Process SME responses:
     - "approved" or similar ‚Üí proceed to grounding document
     - Specific adjustments ‚Üí update the finding with SME context, re-present that finding only, ask again
     - New questions sparked ‚Üí do a lighter research pass (1-2 sources per new hypothesis via tavily_search), present new findings, ask for approval again
   - **Hard gate**: Do not proceed until the SME explicitly approves. Use `AskUserQuestion`: "All findings reviewed. Ready to lock these in and move to curriculum design?" with options "Yes, let's go" and "I want to revisit something"

9. **Grounding Document Output** section (RSRCH-05):
   - Write verified findings to `workspace/{project}/source-material/domain-research-findings.md` (or similar ‚Äî file organization is Claude's discretion per CONTEXT.md)
   - Document format ‚Äî a structured grounding document that downstream stages can parse:
     ```
     # Domain Research Findings: {program_topic}
     Researched: {ISO timestamp}
     Approved by SME: {ISO timestamp}

     ## Research Context
     - **Domain:** {program_topic}
     - **Target audience:** {target_audience}
     - **Skill type:** {skill_type}

     ## Findings

     ### Finding {N}: {Claim summary}
     - **Evidence:** {SUPPORTED|COMPLICATED|CONTRADICTED|GAP}
     - **SME claim:** {original claim}
     - **Evidence summary:** {synthesis}
     - **Sources:** {URLs}
     - **SME response:** {Confirmed|Adjusted: {details}|Added context: {details}}
     - **Implication for curriculum:** {How downstream stages should use this ‚Äî e.g., "SUPPORTED findings are constraints to incorporate", "CONTRADICTED findings become common misconceptions to address", "GAP findings are noted limitations"}
     ```
   - This file lands in source-material/ so downstream stages automatically pick it up via their existing source material loading blocks

10. **Audit Trail** section:
    - Load `.claude/reference/audit-trail-format.md`
    - Read `workspace/{project}/audit-trail.md`. If a "Domain Research" section already exists (re-run), replace it. Otherwise append after Stage 1.
    - Write section following the format:

      ```
      Generated: {ISO timestamp}

      - Curriculum implications derived from evidence patterns

      ### Read but Not Referenced
      {Any sources found but not incorporated into findings. If all were used: "All sources were referenced above."}

      ### SME Confirmation
      - **Confirmed:** {ISO timestamp}
      - **Decision:** Approved domain research findings
      - **Modifications:** {None, or list: "Finding N adjusted: {before} ‚Üí {after}"}
      ```
    - Update Build Summary at top of trail:
      - Add "Stage 1.5: Domain Research" to Stages completed list
      - Update source materials count
      - Update grounding percentage
      - Increment SME checkpoints count
    - Do this silently ‚Äî no announcement to the user.
