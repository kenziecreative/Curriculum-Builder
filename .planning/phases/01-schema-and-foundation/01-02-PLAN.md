**stage-06-metaskills.md:**
- depends_on: stage-04-modules, stage-05-sessions
- output_dir: `05-metaskills/`
- Required fields per metaskill mapping: metaskill_name (enum: [Exploring, Creating, Innovating, Adapting, Imagining, Communicating]), activation_activity (must be a specific named thinking routine — NOT a generic label like "encourage creativity"), transfer_prompt (how this metaskill connects to real-work application), target_sessions (array of session references where this metaskill is activated), evidence_level (enum: [high, moderate, emerging] — high for Exploring/Creating, moderate for Innovating/Adapting, emerging for Imagining)
- Developability hierarchy rule: Exploring and Creating (high evidence) must be activated before Innovating and Adapting in the sequence. This is an ordering constraint, not a suggestion.
- Imagining framing rule: activate through adjacent evidence-based practices (scenario planning, futures thinking, mental simulation). Schema must include evidence_gap_acknowledgment field (boolean) that is required=true for Imagining metaskill.
- Coverage rule: all six metaskills must be activated across the program (not all in every module)
- Distribution rule: no single metaskill should account for more than 30% of total activations
- Duration scaling: 90-min programs activate 2-3 metaskills; semester programs activate all 6 with distribution

**stage-07-transfer.md:**
- depends_on: stage-05-sessions, stage-06-metaskills
- output_dir: `06-transfer/`
- Three required layers, each an object with specific sub-fields:

Pre-program layer:
- readiness_assessment (required)
- manager_briefing_template (required for programs > 4 hours)
- baseline_measurement (required)
- pre_work (required, calibrated to self_direction_level from intake)

In-program layer:
- implementation_intentions (required per module — "When X, I will Y" format)
- real_work_application (required — uses participants' actual work, not hypotheticals)
- error_management_practice (required for open-skill curricula)

Post-program layer:
- spaced_retrieval_schedule (enum intervals: [1_week, 1_month, 3_months])
- spaced_retrieval_prompts (array, one per interval)
- peer_accountability_structure (required)
- manager_checkin_prompts (required for programs > 4 hours)
- community_continuation_design (required — CoP seeding and continuation plan)
- evaluation_design (required object: what, when, by_whom, kirkpatrick_level [enum: Level 1 - Reaction, Level 2 - Learning, Level 3 - Behavior, Level 4 - Results])
- Minimum evaluation level: Level 3 for any program claiming behavior change outcomes

- Skill-type adaptive rule: open-skill curricula get heavier transfer scaffolding than closed-skill
- Warning rule: flag when compressed format + open skills + minimal transfer support are combined
- Duration scaling: 90-min programs get lightweight transfer (pre-work + 1-week follow-up only); semester programs get full ecosystem
- Attachment rule: transfer elements attach to specific sessions and modules (not generic appendices)

**stage-08-marketing.md:**
- depends_on: stage-05-sessions, stage-06-metaskills, stage-07-transfer
- output_dir: `07-marketing/`
- Required fields per marketing element: element_type (enum: [program_description, learning_promise, audience_positioning, testimonial_prompt, enrollment_cta]), content (the marketing text), source_citation (required — reference to specific curriculum element this claim derives from), claim_type (enum: [outcome-based, experience-based, credential-based])
- Traceability constraint: every claim in marketing output must have a source_citation pointing to a specific objective, assessment, or session element. No unsourced claims allowed.
- Learning promises constraint: must reflect actual outcomes from Stage 2 objectives, not aspirational language
- Audience positioning constraint: grounded in intake demographics and expertise data from Stage 1
- Marketing-to-pedagogy ratio rule: marketing output must be < 25% of total curriculum package output (enforce as a validation check, state as a constraint in schema)
- Duration scaling: 90-min programs produce a single program description; semester programs produce full marketing package

**stage-09-validation.md:**
- depends_on: stage-01 through stage-08 (all prior stages)
- output_dir: `08-validation/`
- Output files: schema-report.md, rubric-report.md
- Three-tier validation structure:

Tier 1 — Automated schema checks (required fields):
- field_name, stage, expected_type, actual_value, status (enum: [pass, fail, warning])
- Checks: every objective has bloom_level + paired assessment + transfer task + prerequisite; every session has all 8 TMA fields; every module has social learning with all 4 sub-fields; every metaskill mapping has activation_activity (not label); Bloom's level verbs match claimed levels; sequencing shows prerequisite logic; all 6 metaskills activated across program
- Completion threshold: > 95% of required fields populated to pass

Tier 2 — Rubric-based scoring (confidence scores, NOT pass/fail):
- dimension_name, score (0.0-1.0 confidence), evidence, recommendation
- Dimensions: cognitive_load_appropriateness, scaffolding_coherence, transfer_realism, social_learning_quality, belief_challenging_quality
- These produce confidence scores for human review, not blocking gates

Tier 3 — Human review items:
- item_description, location (specific file and field), review_type (enum: [content_accuracy, contextual_fit, feasibility, quality]), actionable_description (what specifically to evaluate)
- Transfer-specific checks: pre-program readiness fit, manager briefing appropriateness, post-program follow-up feasibility, CoP design appropriateness, evaluation feasibility

- Separation rule: generation and validation must NEVER occur in the same agent call
- Failure reporting: field-level failures with exact stage, file, and field location — not generic messages
- Duration scaling: 90-min programs get Tier 1 only; full programs get all three tiers
  </action>
