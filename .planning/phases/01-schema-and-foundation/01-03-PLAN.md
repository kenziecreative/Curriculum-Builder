**Project CLAUDE.md template:**
Create the project-level CLAUDE.md that will be copied into each workspace project. This is NOT the root CLAUDE.md — it governs generation behavior within a curriculum project. Must stay under 300 lines. Structure:

1. **Project Identity** (5 lines) — "This is a KNZ Curriculum Builder workspace. All generation follows the schemas in `.claude/reference/schemas/`."

2. **Constraint Hierarchy** (10 lines) — State clearly:
   - Schema > Template > Checklist > Inline Directive > Framework Naming > Role Framing
   - When a schema field contradicts a prose instruction, the schema wins
   - When a template structure contradicts a checklist item, the template wins

3. **Pipeline Sequence** (15 lines) — Backward design is the generation sequence, not a suggestion:
   - Stage 1 (Intake) must complete before Stage 2 (Outcomes)
   - Stage 2 (Outcomes) must complete before Stage 3 (Assessments)
   - Stage 3 (Assessments) must complete before Stage 4 (Modules)
   - Stages 4-8 run autonomously after the post-assessment review gate
   - Stage 9 (Validation) runs after all prior stages
   - Never generate content for a stage whose prerequisites are incomplete

4. **Pedagogical Principles** (20 lines, 2-3 sentences each) — The "why" behind the schemas:
   - TMA content arc: ACTIVATE > THEORY > CHECK > METHOD > PRACTICE > APPLICATION > REFLECT > TRANSFER. The arc is inside session frameworks, not a replacement for them.
   - DCR analytical method: Deconstruct > Compare > Rebuild (> Validate for wicked domains). Scaffolded for novices.
   - Transfer design: pre/in/post-program. Transfer is the mechanism that makes everything else matter. Not optional.
   - Social learning: required per module with genuine interdependence, not "discuss with a partner."
   - Backward design: outcomes drive assessments drive content. Never the reverse.

5. **Schema References** (5 lines) — "All schemas are in `.claude/reference/schemas/`. Each generation command MUST include the relevant stage schema as context. Generate output matching the schema. All required fields must be present. Enum fields must use exact values listed."

6. **State Management** (10 lines):
   - Read STATE.md at session start (the hook handles this)
   - Update STATE.md after every stage transition
   - NEVER announce "updating STATE.md" — state updates are silent (INFR-10)
   - STATE.md is the single source of truth for pipeline position

7. **Never** section (15 lines, 5-8 hard prohibitions):
   - Never generate content without the relevant stage schema loaded as context
   - Never skip transfer_connection in session design — it is a required field, not optional enrichment
   - Never use generic reflection prompts ("what did you learn?") — reflection must be specific to the session's content and objectives
   - Never create social learning activities without genuine interdependence structure
   - Never advance past a review gate without explicit user approval
   - Never write to a future-stage directory when the preceding stage is incomplete
   - Never announce state updates — update STATE.md silently

Total target: 200-250 lines (well under the 300-line cap).

Create a session-start hook at `.claude/hooks/session-start.md`. Note: there are already hooks in `.claude/hooks/` (milestone-check.sh, pre-commit-secrets.sh, pre-compact-check.sh) — this is a new addition, not a replacement.

The session-start hook is a Claude Code hook that runs when a new session begins. It should:

1. **Check for active curriculum project** — Look for `workspace/*/STATE.md`. If no workspace exists or no STATE.md is found, do nothing (the user hasn't initialized a project yet).

2. **Read STATE.md** — Parse the current stage, status, completed stages, key decisions, and session continuity fields.

3. **Surface context silently** — Display to the user (but do NOT announce "I'm reading STATE.md" per INFR-10):
   - Current pipeline stage and status
   - Last session date and what was stopped at
   - Next action to take
   - Any review gates that are pending approval
   - A brief summary of key decisions from intake (if intake is complete)

4. **Block advancement without resolution** — If the session continuity section shows a "Stopped At" value that indicates mid-stage work, the hook should note that the previous session was interrupted and the user should resolve the current stage before advancing.

The hook format should follow Claude Code hook conventions. Since the existing hooks in `.claude/hooks/` are shell scripts, and this is a markdown-based hook description (for Claude Code's hook system, not git hooks), use the appropriate format. The hook is a reference file that instructs Claude's behavior at session start — it describes what Claude should do, not executable code.

Key rules encoded in the hook:
- Silent state reading (INFR-10) — never say "Let me check STATE.md" or "Updating STATE.md"
- Resume awareness — surface what was happening and what's next
- Gate awareness — if a review gate is pending, remind the user
- Graceful absence — if no project exists, skip silently

Note: The hook file name is `session-start.md` to indicate it fires on session start. It will be referenced by the plugin's hook configuration.
