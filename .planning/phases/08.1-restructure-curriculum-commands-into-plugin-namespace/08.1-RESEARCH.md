## Summary

Claude Code plugins are directories with a `.claude-plugin/plugin.json` manifest at the plugin root. Commands, agents, and skills all sit at the plugin root (NOT inside `.claude-plugin/`). The plugin's `name` field in `plugin.json` becomes the command namespace: a command file at `commands/intake.md` in a plugin named `curriculum` is invoked as `/curriculum:intake`.

The proposed location `.claude/plugins/curriculum/` places the plugin inside the project's `.claude/` directory. This is non-standard — plugins are normally standalone directories — but it is a valid structure that Claude Code can load via `--plugin-dir`. The `enabledPlugins` key in `.claude/settings.json` uses a `"name@marketplace"` format and requires a marketplace registration, which is complex for a project-bundled plugin. The simplest verified approach for project-local activation is a local-scoped marketplace registration pointing at the plugin directory.

The core migration work is mechanical: 12 command files move and are renamed, 2 agent files move, and `/knz-*` references throughout command bodies, hook scripts, and `session-start.md` are updated to `/curriculum:*`. The pre-tool-use.sh hook also hardcodes `/knz-*` command names in its error messages and needs updating.

**Primary recommendation:** Create the plugin at `.claude/plugins/curriculum/` with `.claude-plugin/plugin.json` declaring `name: "curriculum"`. Use `extraKnownMarketplaces` with `directory` source in `.claude/settings.json` to register it at the project level, then `enabledPlugins` to activate it. Test with `--plugin-dir .claude/plugins/curriculum` during development.

## Standard Stack

### Core
| Component | Format | Purpose | Notes |
|-----------|--------|---------|-------|
| `plugin.json` | JSON | Plugin manifest — name, version, description, author | Lives at `.claude-plugin/plugin.json` inside plugin root |
| `commands/` | Markdown with YAML frontmatter | User-invokable commands | At plugin root; namespaced by plugin name |
| `agents/` | Markdown with YAML frontmatter | Subagents dispatched via `Task` tool | At plugin root |
| `settings.json` | JSON | Plugin default settings; only `agent` key currently supported | At plugin root |

### plugin.json Fields
| Field | Required | Purpose |
|-------|----------|---------|
| `name` | YES | Namespace for all commands. `curriculum` → `/curriculum:*` |
| `version` | Recommended | Semantic version string |
| `description` | Recommended | Shown in plugin manager |
| `author` | Optional | `{ "name": "...", "email": "..." }` |
| `hooks` | Optional | Path to hooks.json if plugin has its own hooks |

**Minimal valid plugin.json:**
```json
{
  "name": "curriculum",
  "version": "1.0.0",
  "description": "Curriculum builder plugin — encodes pedagogical doctrine as structural constraints to produce delivery-ready curriculum"
}
```

### Plugin Activation: How Claude Code Finds the Plugin

There are three verified mechanisms:

**Option A: `--plugin-dir` flag (development/testing only)**
```bash
claude --plugin-dir .claude/plugins/curriculum
```
Works session-by-session. Good for testing during migration. Does NOT persist.

**Option B: Local marketplace registration + enabledPlugins (project-bundled, recommended)**

In `.claude/settings.json`:
```json
{
  "extraKnownMarketplaces": {
    "curriculum-local": {
      "source": {
        "source": "directory",
        "path": ".claude/plugins/curriculum"
      }
    }
  },
  "enabledPlugins": {
    "curriculum@curriculum-local": true
  }
}
```

This registers the plugin as a local marketplace entry and enables it at project scope. When a collaborator opens the project, Claude Code prompts them to install the marketplace/plugin.

**Option C: User-level install (not project-bundled)**
```
/plugin install curriculum@local
```
Only works after user manually adds the local marketplace. Doesn't travel with the repo.

**Recommendation:** Use Option B (extraKnownMarketplaces + enabledPlugins) in `.claude/settings.json`. It travels with the repo and requires no manual steps from collaborators.

### Known Issue: enabledPlugins Merge Bug

**Source:** GitHub Issue #27247 and #25086

If `enabledPlugins` is set only in `settings.local.json` but NOT also present as a key in `settings.json`, it is silently ignored. The fix is to ensure `enabledPlugins` appears in `settings.json` (even if empty or with the project plugin) before per-machine overrides in `settings.local.json` apply.

**Mitigation:** Put `enabledPlugins` with the curriculum plugin entry directly in `.claude/settings.json` (committed to git) to avoid the merge-bug trap.

### Command File Format

Command files inside `commands/` should have YAML frontmatter with at minimum a `name` field. The `name` must match the filename stem and is used by Claude Code to register the command:

```markdown
---
name: curriculum:intake
description: Run structured curriculum intake — new programs or audit mode for existing materials
---

# /curriculum:intake

[command body...]
```

The existing command files have plain H1 headings (e.g., `# /knz-intake`) rather than YAML frontmatter. Both formats work — Claude Code reads the filename for registration. However, adding YAML frontmatter with `name` and `description` is the recommended current pattern (verified in foresight, research-scaffold plugins).

For this migration, updating the H1 heading from `# /knz-intake` to `# /curriculum:intake` is sufficient to keep them consistent with command invocation — full YAML frontmatter conversion is deferred per the "not converting to skills format" decision.

## Common Pitfalls

### Pitfall 1: Putting commands inside `.claude-plugin/`
**What goes wrong:** Commands placed at `.claude-plugin/commands/` instead of `commands/` at the plugin root are invisible to Claude Code.
**Why it happens:** The `.claude-plugin/` directory looks like the "right place" to put plugin content.
**How to avoid:** The official docs are explicit: only `plugin.json` goes in `.claude-plugin/`. Everything else goes at the plugin root.
**Warning signs:** `/curriculum:intake` returns "unknown command" after migration.

### Pitfall 2: Dangling `/knz-*` references
**What goes wrong:** User runs `/curriculum:intake` and sees a user-facing message saying "Run `/knz-outcomes` to continue" — breaks the UX narrative.
**Why it happens:** The reference audit is exhaustive — it's easy to miss one occurrence buried in a long command file (knz-intake.md is 675+ lines).
**How to avoid:** After all file moves and edits, run a final grep across the entire `.claude/` directory for any remaining `/knz-` strings.
**Verification command:**
```bash
grep -rn "/knz-" .claude/
```
Expected result: zero matches.

### Pitfall 3: The enabledPlugins merge bug
**What goes wrong:** `enabledPlugins` is added to `settings.local.json` only, not `settings.json`. The plugin silently doesn't load.
**Why it happens:** Documented Claude Code bug (GitHub issues #27247, #25086) — `enabledPlugins` in `settings.local.json` is ignored unless the key also exists in `settings.json`.
**How to avoid:** Put `enabledPlugins` with the curriculum entry directly in `.claude/settings.json`.

### Pitfall 4: Forgetting the `pre-tool-use.sh` hook
**What goes wrong:** The hook still references `/knz-intake` etc. in its blocking error messages. Users see confusing messages telling them to run a command that no longer exists.
**Why it happens:** Hook files are shell scripts, not markdown — easy to overlook in a "markdown migration."
**How to avoid:** The hook at `.claude/hooks/pre-tool-use.sh` has 7 PREREQ_CMD values that all need updating.

### Pitfall 5: Partial state — old files remain alongside new ones
**What goes wrong:** Old `knz-*.md` files stay in `.claude/commands/`. Claude Code sees BOTH `/knz-intake` and `/curriculum:intake`. Users may accidentally run the old commands; confusion ensues.
**How to avoid:** Delete all 12 old `knz-*.md` command files from `.claude/commands/` as part of the same operation that creates the new plugin files. Do not leave them as "backup copies."

### Pitfall 6: Agents not found by Task dispatches
**What goes wrong:** Commands that use `Task` to dispatch `knz-validator` or `session-generator` may reference the agent by name. If the agent files move but the name reference in the dispatching command is tied to the old path, Task may fail silently.
**Why it happens:** `Task` dispatches agents by name — the name declared in the agent file's frontmatter, not the file path.
**How to avoid:** Verify the agent frontmatter `name:` field and cross-check that dispatching commands reference the same name. The move itself doesn't change names, but confirm nothing in the command files refers to agents by their old path.

## Open Questions

1. **Plugin activation for a plugin inside `.claude/`**
   - What we know: Official docs show plugins as standalone directories. `extraKnownMarketplaces` with `directory` source supports local paths. `--plugin-dir` works for dev/testing.
   - What's unclear: Whether a path like `.claude/plugins/curriculum` (inside the project's `.claude/`) is treated differently from a path outside it.
   - Recommendation: Test with `--plugin-dir .claude/plugins/curriculum` immediately after creating the structure. If it loads, the location is valid. If not, move to a sibling directory (e.g., `curriculum-plugin/` at repo root) — but this seems unlikely to be necessary given the official docs support any directory path.

2. **`extraKnownMarketplaces` with `directory` source for project-bundled use**
   - What we know: The `directory` source type exists in the marketplace source type list.
   - What's unclear: Whether a relative path (`.claude/plugins/curriculum`) works or if an absolute path is required.
   - Recommendation: Use a path relative to the project root (`".claude/plugins/curriculum"`). If that fails, switch to `$CLAUDE_PROJECT_DIR/.claude/plugins/curriculum` or verify via testing. The `--plugin-dir` fallback is always available.
