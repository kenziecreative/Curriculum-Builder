  <action>
Create the /knz-intake command file. This is the largest and most behaviorally complex command in the plugin. Follow the sanity-check.md pattern (frontmatter + behavior sections) but with significantly more behavioral specification.

**Structure the command in these sections:**

## Prerequisites
- Check workspace/*/STATE.md exists (if not, tell user to run /knz-init)
- Check Stage 1 status is "not-started" or "in-progress" (if "complete", tell user intake is done and suggest /knz-resume)
- If "in-progress", load previously captured fields from STATE.md Key Decisions and skip completed thematic groups

## Persona
Maintain an expert instructional designer persona throughout (INFR-11). Tone: warm, knowledgeable, curious — like a consultant who genuinely wants to understand the program. Never use instructional design vocabulary with the user. Internally, the command knows it's mapping answers to schema fields; externally, it's just having a good conversation about their program.

## Opening
Brief, warm context-setting (2-3 sentences max): explain that you'll ask a few questions about their program and learners to shape the curriculum. Set expectations: "I'll ask in a few short rounds — about your learners, your program structure, and what success looks like."

## Thematic Group 1: About Your Learners (maps to target_audience + self_direction_level)
Ask 2-3 questions in natural conversation:
- "Who are the people taking this program? What do they do day-to-day?" → maps to target_audience.description
- "What can they already do related to this topic, and what can't they do yet?" → maps to target_audience.prior_knowledge (behavioral format per INTK-03). If the user gives labels like "beginners" or "intermediate", push back: "That helps — can you give me a concrete example? Like, can they already [do a basic version of the skill], or are they starting from scratch?"
- "How comfortable are your learners with learning on their own versus needing step-by-step guidance?" → maps to self_direction_level. The command internally maps the answer to the Grow model enum. Do NOT expose the enum values or Grow model name to the user.

**Pushback rules for Group 1 (INTK-06):**
- If prior_knowledge is vague ("they're new to this"): reframe as "Help me understand the gap — what's a task they can handle today, and what's a task that would trip them up?"
- If audience description is too broad ("business owners"): ask "What kind of business owners? Solo freelancers, small teams of 5-10, funded startups? The program will be different for each."

After Group 1 answers are confirmed, write to STATE.md Key Decisions: Audience, Expertise Level, Self-Direction Level. Update Stage 1 status to "in-progress" and Stopped At to "Completed learner questions".

## Thematic Group 2: About Your Program (maps to program_topic + duration_and_format + skill_type)
Ask 2-3 questions:
- "What's the core topic or skill this program teaches?" → maps to program_topic. Push back on broad topics: "That's a big area — is there a specific slice? For example, 'negotiating supplier contracts' rather than 'business negotiation skills'."
- "How long is the program, and how is it delivered? For example: a single 3-hour workshop, a 6-week series of 90-minute sessions, or something else?" → maps to duration_and_format fields. The command extracts delivery_mode, modality, contact_hours, session_count, and session_length_minutes from the natural-language answer. Ask follow-up if ambiguous (e.g., "Is that in-person, virtual, or a mix?")
- "Is there a right way to do this skill, or does it depend on judgment and context? For example, operating equipment has a correct procedure, but giving feedback depends on the situation." → maps to skill_type (closed/open). Do NOT use the terms "closed skill" or "open skill" with the user.

**Conditional: cultural_orientation** — If contact_hours >= 2, add: "Will your learners mostly be from backgrounds that value individual achievement, group harmony, or a mix?" This is conversational, not clinical. Map to the enum internally.

**Conditional: extended fields** — If contact_hours >= 16, add: "About how many people will be in each cohort?" and "Are there any programs they should have completed before this one?"

After Group 2 answers are confirmed, write to STATE.md Key Decisions: Duration, Skill Type, Cultural Orientation (if applicable). Update Stopped At to "Completed program questions".

## Thematic Group 3: About Success (maps to success_criteria + transfer_context)
Ask 2 questions:
- "After this program, where will learners actually use what they learned? Give me a specific work situation — a meeting, a client interaction, a daily task." → maps to transfer_context. Push back if abstract: "Can you make that more concrete? Instead of 'in their jobs,' something like 'when they're sitting across from a new supplier and need to negotiate terms.'"
- "How will you or your organization know this program worked? What would you see people doing differently?" → maps to success_criteria. Push back on vague success: "I hear you — but 'better understanding' is hard to measure. What would you actually see them doing? Completing something faster? Handling a situation they used to avoid?"

**Pushback rules for Group 3 (INTK-06):**
- success_criteria must use observable verbs. If user says "understand" or "appreciate", reframe: "Those are great internal goals. Externally, what would change? What would someone watching them notice?"
- transfer_context must name a specific situation, not a domain. "At work" → push for the specific moment.

After Group 3 answers are confirmed, write to STATE.md Key Decisions: Transfer Context. Update Stopped At to "Completed success questions".

## Confirmation and Gate (INTK-05 + PIPE-04 + INFR-05)
After all three groups are complete:

1. Present a structured summary of everything captured, using the user's own words (not schema field names). Format as a clean, readable block:

   **Your Program at a Glance**
   - **Topic:** {program_topic}
   - **Learners:** {target_audience.description}
   - **Current skill level:** {prior_knowledge in plain language}
   - **Program format:** {contact_hours} hours, {session_count} sessions, {modality}, {delivery_mode}
   - **Where they'll use it:** {transfer_context}
   - **Success looks like:** {success_criteria}

2. Use AskUserQuestion (INFR-06) with three options:
   - "Looks good — let's keep going"
   - "I want to edit something"
   - "Start over from scratch"

3. On "Looks good":
   - Write complete project-brief.md to workspace/{project-name}/00-project-brief/project-brief.md matching the stage-01-intake schema exactly (include schema as generation context)
   - Update STATE.md: Stage 1 status to "complete", Post-Intake gate to "approved", Next Action to "Run /knz-outcomes to begin outcome design" (even though that command doesn't exist yet)
   - End with a brief forward-looking message: "Great — your program brief is locked in. Next up is designing learning outcomes, which I'll build from what you've told me."

4. On "Edit something": ask what they want to change, update the relevant field, re-present summary, re-ask.

5. On "Start over": reset all Key Decisions to "—", set Stage 1 to "not-started", restart from Group 1.

## context_of_use Derivation
The schema requires target_audience.context_of_use. This can be derived from transfer_context + target_audience.description without asking a separate question. When writing project-brief.md, synthesize this field from the answers already captured.

## Schema Compliance
When writing project-brief.md, include the stage-01-intake.md schema as generation context. The output file must contain ALL required fields with exact enum values. The command file should reference the schema path: .claude/reference/schemas/stage-01-intake.md

## State Management
All STATE.md writes are silent per INFR-10. Never say "Let me save your progress" or "Updating your state." Just do it between conversation turns.
