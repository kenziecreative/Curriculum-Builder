**knz-resume.md:**

Create the /knz-resume command. This is the generic "where am I?" command that works for all stages from Phase 2 onward. Behavior:

1. Find active project: look for workspace/*/STATE.md. If multiple projects exist, use AskUserQuestion to let user pick. If none exist, tell user to run /knz-init.
2. Read STATE.md and extract: Current Stage, Stage Progress table, Key Decisions (if intake complete), Review Gates, Session Continuity fields.
3. Display a scannable status block:
   - Project name (from directory name, title-cased)
   - Current stage name and status (e.g., "Stage 1: Intake — in progress")
   - If interrupted: what was last done (Stopped At field)
   - Next action (Next Action field from Session Continuity)
   - If a review gate is pending: surface it prominently
   - If intake is complete: one-line summary of key decisions (duration, audience, expertise)
4. Route to the right command: suggest the appropriate /knz-{stage} command or /knz-approve if a gate is pending.
5. For stages that don't have commands yet (Stages 2-9 in Phase 2), show: "Stage {N} ({Name}) — command available in a future update. Run /knz-resume to check back."

Format: Clean, scannable, no walls of text. Use bold for the project name and stage. Use a simple table for stage progress only if the user asks for detail.

Silent state management per INFR-10.

**knz-approve.md:**

Create the /knz-approve command. This is the generic stage gate approval command used for review gates that are NOT inline (post-assessment gate in Phase 3, final validation gate in Phase 7). The post-intake gate is handled inline by /knz-intake itself.

Behavior:
1. Find active project (same logic as resume).
2. Read STATE.md Review Gates table. Find any gate with status "pending".
3. If no gate is pending: tell user there's nothing to approve right now and show current position.
4. If a gate is pending:
   a. Read the relevant stage output (e.g., for post-assessment gate, read files in 02-assessments/).
   b. Present a condensed summary of what was produced — key outputs, counts, notable decisions.
   c. Use AskUserQuestion with three options per INFR-06:
      - "Approve and continue" — marks gate as approved, advances pipeline
      - "I have concerns" — asks user to describe what needs changing, does NOT approve
      - "Start this stage over" — resets the stage to not-started, clears output
   d. On approval: update Review Gates table status to "approved", update Session Continuity with next action.
5. Silent state management per INFR-10.

The approval summary should be written in plain language — no schema field names, no instructional design jargon unless the user used it during intake.
