    Same pattern as other commands. Stop with appropriate message if no project found.

    Read and check files in the workspace. Collect issues as a list of {file (relative path), plain-language description, fix command}.

    **Check A — Missing required stage files:**
    Check for existence of each required file/directory listed in the interfaces section. For each missing item, add an issue with the relevant fix command:

    - Missing 06-transfer/ or empty → "Not yet generated — run /curriculum:transfer"
    - Missing 07-marketing/ or empty → "Not yet generated — run /curriculum:marketing"

    **Check B — Content issues in generated output files:**
    For each .md file in 04-sessions/*/, 07-marketing/, and 06-transfer/:
    - NEEDS: markers: scan for any line starting with "NEEDS:" → "has unresolved content markers — run {stage command} to regenerate"
    - TMA phase labels as headers: scan for lines matching a heading followed by ACTIVATE, THEORY, METHOD, APPLICATION, BRIDGE, or TRANSFER as the full heading text → "contains formatting labels that should not appear in facilitator guides — run /curriculum:sessions to regenerate"

    Do NOT re-run vocabulary audit (Phase 13 handles that). Checks B is a slip-through net, not a full audit.

    If no issues: "Your curriculum is ready to deliver — no issues found."

    If issues exist: Show count + list in the format from the interfaces section. Each item: file path (relative to project root), plain-language description on one line, fix command indented below.

    Do NOT show: check IDs, technical patterns, binary pass/fail without specifics, any offer to fix automatically.

    **Warm handoff close (standalone use only):**
    If called standalone and passes: "Run `/curriculum:approve` to mark your curriculum delivery-ready."
    If called standalone and fails: the fix commands in the issue list are the next step — no additional close needed.

    **Note on silent invocation from approve:**
    verify.md documents that it may be invoked by approve.md using the Task tool. In that mode, verify returns a structured result rather than printing to the user — approve.md handles presentation. Verify does not need to detect which mode it is in; approve.md controls the invocation.
