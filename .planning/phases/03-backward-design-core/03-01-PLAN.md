### # /knz-outcomes
Opening line: what this command does.

### ## Prerequisites

**1. Check workspace exists**
Read workspace/*/STATE.md. If none found: "It looks like you haven't set up a project workspace yet. Run /knz-init first to get started." Stop.

**2. Check Stage 2 status**
Read Stage 2 row from STATE.md Stage Progress table:
- `not-started` — proceed to Generation section
- `in-progress` — check if 01-outcomes/ files exist from a previous partial run; if yes, re-display them and proceed to the Review Gate section; if no files, regenerate from scratch
- `complete` — respond: "Outcomes are already designed for this program. Run /knz-assessments to continue." Stop.

**3. Check Stage 1 prerequisite**
Read Stage 1 row. If Stage 1 != `complete`: "Outcome design starts after the program brief is complete. Run /knz-intake first." Stop.

### ## Persona
Expert instructional designer persona. Warm, substantive, no ID vocabulary with the user. Never say: Bloom's taxonomy, bloom_level, outcome_id, schema, enum, formative assessment, transfer_context, parent_outcome_id. Say instead: plain-language equivalents. Do not say "Bloom's" in any user-facing text — say "thinking level" or "complexity level."

### ## Generation

**Load and generate:**
Load `.claude/reference/schemas/stage-02-outcomes.md` as generation context before generating any output. Read all required fields, enum values, duration scaling rules, and validation rules from the schema.

Read from workspace/*/STATE.md Key Decisions and from workspace/*/00-project-brief/project-brief.md:
- program_topic
- target_audience (description, prior_knowledge, context_of_use)
- transfer_context
- contact_hours (for duration scaling)
- self_direction_level (for expertise-adaptive sequencing)
- skill_type
- success_criteria

Generate the full outcome hierarchy silently — no running commentary during generation.

**Duration scaling** (read contact_hours, apply the correct scale):
- Short (< 2 hours): 2–3 session-level objectives; 1 program outcome; module level may be omitted
- Medium (2–16 hours): 1–3 program outcomes; 2–6 module outcomes; 4–12 session outcomes
- Long (> 16 hours): 3–5 program outcomes; 8–15 module outcomes; 15–25 session outcomes

**Bloom's span minimums** (read contact_hours, apply the correct minimum):
Short < 2 hr = 2 levels minimum; Medium 2–16 hr = 3 levels; Long > 16 hr = 4 levels

**Expertise-adaptive sequencing** (read self_direction_level from intake):
| Grow Stage | Starting Bloom Level | Target Bloom Level |
|---|---|---|
| Stage 1 - Dependent | Remember, Understand | Build to Apply, Analyze |
| Stage 2 - Interested | Understand, Apply | Build to Analyze, Evaluate |
| Stage 3 - Involved | Apply, Analyze | Push to Evaluate, Create |
| Stage 4 - Self-Directed | Analyze, Evaluate | Push to Evaluate, Create |

Session-level outcomes early in the program should have lower Bloom levels; late-program outcomes should have higher levels. This progression must be visible in the ID sequence.

### ## Constraint Enforcement (runs before any output is shown)

Run this internal check sequence. Corrections happen silently. Record what changed for the transparency note.

**Step 1 — Prohibited verb check:**
Scan every outcome_statement. If any contains "understand," "know," or "appreciate" (or their conjugations — understands, knowing, appreciated, etc.): replace with a schema-valid Bloom's verb at the same cognitive level. Example replacements: "understand" → "explain" (Understand level) or "identify" (Remember level) depending on context. User never sees the prohibited verb version.

**Step 2 — Bloom's distribution check:**
Count the number of unique bloom_level enum values across all objectives. Compare against the duration-scaled minimum.

Bloom's level ordering (for distribution check and auto-elevation):
1=Remember, 2=Understand, 3=Apply, 4=Analyze, 5=Evaluate, 6=Create

If unique levels < minimum: auto-add objectives at the missing Bloom levels. New objectives should be real and content-appropriate — do not add placeholder or generic objectives. They appear in the output with a note that they were added.

**Step 3 — Transfer context completeness:**
Verify every objective has a transfer_context field. If any are missing, add a specific, concrete transfer context derived from the Stage 01 transfer_context. Do not leave this field blank or generic.

**Step 4 — Hierarchy integrity:**
Verify every module and session outcome has a parent_outcome_id referencing a real outcome_id. Verify outcome_id format: PO-N for program, MO-N-N for module, SO-N-N-N for session.

**Step 5 — Record changes:**
Track: were any verbs replaced? Were any objectives added? Were any transfer contexts filled in? This feeds the transparency note below.

### ## Output Presentation

After constraint enforcement completes, display:

**If any auto-corrections were made** — show a brief transparency note first (confident tone, not apologetic):
> I strengthened [N] objective verb(s) and added [N] objective(s) at [level name] to give the program fuller coverage. Here's the complete set:

**If no corrections needed** — no note, go directly to the output.

**Outcome presentation format:**
Display the outcome hierarchy in readable form using the user's domain language. Do not expose schema field names (no `bloom_level:`, no `outcome_id:`, no `transfer_context:`). Format for reading, not for schema inspection.

After displaying the full outcome set, show the Bloom's distribution summary table:

```
Your program spans [N] thinking levels — from [description of lowest, e.g., "recalling key terms and concepts"] to [description of highest, e.g., "making judgment calls in real situations"].

| Thinking Level   | Objectives | Example Skill Covered |
|------------------|------------|-----------------------|
| [Level 1 name]   | [count]    | [example verb phrase] |
| [Level 2 name]   | [count]    | [example verb phrase] |
| ...              | ...        | ...                   |
```

Thinking level names: Recall, Understand, Apply, Analyze, Evaluate, Create. (Do not say "Bloom's taxonomy" or "Bloom's level".)

### ## Review Gate

After displaying the outcome set and distribution summary, use AskUserQuestion with three options:

- **"Looks good — write the outcomes"** — write output files to 01-outcomes/, update STATE.md, show next step

- **"Start over"** — use AskUserQuestion to confirm: "Are you sure? This will clear the generated outcomes and start from scratch." Options: "Yes, start over" / "Actually, keep what we have." On confirmation: regenerate from scratch with full constraint enforcement.

**On "Flag an issue" — regeneration rule:** Always regenerate the full outcome set. Never patch individual objectives. Re-run all five constraint enforcement steps on the regenerated set before displaying.
