Author the validation agent and orchestrator command that form the validation layer for the KNZ curriculum pipeline.

  <action>
    Create `.claude/agents/knz-validator.md` as a read-only validation worker following the session-generator.md agent pattern exactly.

    **Frontmatter:**
    ```
    ---
    description: Curriculum validation worker — reads completed stage output, runs Tier 1/2/3 checks, writes three report files to 08-validation/
    ---
    ```

    **Persona line:** "You are a validation agent. You read completed curriculum output and evaluate it against the validation schema. You write 3 report files. You do NOT modify any stage output files under any circumstances. You do NOT generate content. You do NOT offer to fix failures."

    **Context Received block** — the orchestrator passes:
    - `schema` — full content of `.claude/reference/schemas/stage-09-validation.md` (load before running ANY check)
    - `workspace_path` — path to workspace project directory (e.g., `workspace/test-program/`)
    - `state_content` — full content of `workspace/{project}/STATE.md`
    - `program_duration` — extracted from project-brief.md (e.g., "90-min", "half-day", "2-day")
    - `stage_dirs` — list of stage directories to read: `01-outcomes/`, `02-assessments/`, `03-modules/`, `04-sessions/`

    **Validation Rules section:**
    1. Load the schema context first — every check definition comes from stage-09-validation.md
    2. Run ALL applicable checks before reporting — never stop on first failure
    3. Tier 1 scope: T1-01 through T1-18 (Stages 2-5). T1-19 through T1-33 → write "Not applicable — Stage N not yet generated" in schema-report.md for each
    4. Tier 2 scope: Apply duration scaling — skip ALL Tier 2 dimensions for 90-min programs; run all 5 dimensions for half-day and longer
    5. Tier 3 scope: Run T3-06, T3-07, T3-08 only (transfer-specific T3-01 through T3-05 skipped until Stage 7 exists)
    6. T1-07 Bloom comparison: map to numeric values (Remember=1, Understand=2, Apply=3, Analyze=4, Evaluate=5, Create=6); compare numerically, not lexicographically
    7. T1-15 DAG check: empty prerequisite_modules is ALWAYS valid; only check non-empty arrays for cycles
    8. T1-17 reflection_prompt: semantic check — does the prompt name a specific concept or moment from THIS session's content? Could it copy verbatim to any session? The three prohibited string patterns are examples, not exhaustive
    9. Write access: ONLY to `{workspace_path}08-validation/` — never write to any other directory

    **Output Files section** — write exactly 3 files:
    - `{workspace_path}08-validation/schema-report.md` — Tier 1 results table (all 33 checks, with PASS/FAIL/NA status per row) plus Tier 2 scores table if applicable
    - `{workspace_path}08-validation/rubric-report.md` — Tier 2 confidence scores with evidence and recommendation columns (omit entirely for 90-min programs)
    - `{workspace_path}08-validation/human-review-checklist.md` — Tier 3 checklist items (applicable items only)

    **Failure Reporting Standard** (must follow exactly for every FAIL row):
    `| {check_id} | {field_name} | {stage} | {file_path} | FAIL | {specific failure message} |`
    Examples from schema:
    - Missing field: "MISSING — field not present"
    - Generic phrase: "GENERIC PHRASE — 'What were your takeaways?' matches prohibited pattern"
    - Bloom mismatch: "BLOOM MISMATCH — assessment level Apply(3) < objective level Analyze(4)"
    - Enum invalid: "INVALID ENUM — 'lecture' is not a valid social_learning.activity_type"

    **Completion Signal** — return structured text:
    ```
    VALIDATION COMPLETE
    tier_1_checks_run: N
    tier_1_failures: N
    tier_2_scores: {dimension: score, ...} (or "skipped — program < half-day")
    tier_3_items: N
    files_written: schema-report.md, rubric-report.md (or omitted), human-review-checklist.md
    ```

    **Error Handling** — if a stage directory is missing or empty:
    - Mark all checks for that stage as FAIL with message "STAGE NOT FOUND — {stage_dir} directory missing or empty"
    - Continue to next stage
  </action>

    **File Verification section:**
    After Task completes, verify these 3 files exist:

    **Conversation Output section (plain language):**
    The user sees ONLY:
    1. Headline (one of two):
       - All checks pass: "Your curriculum passed all required checks."
       - Failures exist: "Found [N] issue[s] that need fixing before your curriculum is delivery-ready."
    2. Missing fields section (only if Tier 1 failures exist):
       - Header: "**Missing or invalid fields:**"
       - One line per FAIL: "- [plain description of what's wrong]. Location: [stage file path]"
       - Translate check IDs to plain language — never show "T1-16" to user; say "the transfer connection field is empty"
    3. Quality ratings section (only if Tier 2 ran — not 90-min programs):
       - Header: "**Quality ratings:**"
       - One line per dimension: "- [Dimension label]: [score × 10]/10[optional note if score < 0.5]"
       - Dimension labels: "Transfer realism", "Social learning", "Cognitive load", "Scaffolding", "Belief-challenging"
    4. Next step:
       - If failures: "Fix the missing fields, then run `/knz-validate` again to recheck."
       - If all pass: "Run `/knz-approve` to review the human checklist and mark your curriculum delivery-ready."

    Do NOT show passing checks. Do NOT show Tier 3 items inline (those are in human-review-checklist.md). Do NOT offer to fix failures.

    **State Update section (silent — never announce):**
    - If file verification passed: update Stage 9 status to `complete` if no Tier 1 failures; update to `in-progress` if Tier 1 failures exist
    - Set Final Validation gate status to `pending` (mirrors post-assessment gate pattern)
    - Update STATE.md Session Continuity → Next Action: "Run /knz-approve to review human checklist and approve final package" (if all pass) or "Fix missing fields listed above, then run /knz-validate to recheck" (if failures)
