<objective>
Author two files — `session-generator.md` (per-module subagent worker) and `knz-sessions.md` (orchestrator command) — that together enable parallel session generation across all modules without accumulating session content in the main context window.

Purpose: Session content is the largest volume output in the pipeline — 4 files per session across all modules. Without parallel subagents, generating this sequentially would collapse the main context window on programs with 3+ modules. The subagent architecture (one Task per module) keeps each module's session content isolated to its own worker context. The orchestrator manages dispatch, progress, and verification without holding content.

Output:
- `.claude/agents/session-generator.md` — the worker that generates sessions for one module
- `.claude/commands/knz-sessions.md` — the orchestrator that dispatches workers and assembles results
</objective>

**File header:**
```yaml
---
description: Per-module session generation worker — generates all sessions for one module including TMA arc, DCR scaffolding, facilitator guides, participant materials, and slide outlines
---
```

**Section structure to implement:**

**## Context Received**
List what the orchestrator provides:
- module_spec: Full content of 03-modules/M-N/module-spec.md (module name, objectives, session count, content chunks, social_learning, primary_metaskill, belief_challenging_encounter)
- outcomes: Full content of 01-outcomes/learning-objectives.md (all SO-* level outcome_ids for this module)
- project_brief: Full content of 00-project-brief/project-brief.md (skill_type, self_direction_level, contact_hours, modality, transfer_context)
- schema: Full content of .claude/reference/schemas/stage-05-sessions.md
- output_dir: The 04-sessions/ path to write files to (e.g., workspace/test-program/04-sessions/)

**## Generation Rules**

Load stage-05-sessions.md schema as generation context. Generate all sessions for this module.

Session count: derived from module_spec sessions_planned count. Each session gets a session_id in M-N-S-N format.

**Session template selection (apply in precedence order for each session):**
1. If skill_type == closed AND session bloom_level in [Remember, Understand, Apply] → `gagne`
2. If session is structured around a problem to solve AND bloom_level >= Apply → `merrill`
3. If session opens with learner investigation before direct instruction → `5e_7e`
4. If program context is professional development / coaching → `wippea`
5. Default → `universal_tma_arc`
For Stage 1-2 self_direction_level: prefer `gagne` or `universal_tma_arc` — inquiry-first templates require sufficient prior knowledge.

**TMA Arc requirement (all 8 fields, every session — no exceptions):**

For each session, generate:

1. `prior_knowledge_activation` (ACTIVATE) — specific activity connecting to this session's content; target_prior_knowledge names specific knowledge (not "prior learning"); duration_minutes included

2. `learning_objectives` (THEORY setup) — SO-* outcome_ids from this module that this session addresses

3. `content_chunks` (THEORY) — 2-4 chunks for Stage 1-2 learners; up to 6-7 for Stage 3-4; each chunk has chunk_name, key_concepts array (2-4 concepts for novices), instructional_method, duration_minutes

4. `formative_check` (CHECK) — occurs after content_chunks, before guided_practice; check_activity describes what learners do; feedback_mechanism describes how results are used by facilitator or learners

5. `guided_practice` (METHOD) — active learner production (not passive); scaffolding_provided names specific structures; facilitator_role describes observation and feedback actions; duration_minutes

6. `independent_practice` (PRACTICE) — reduced scaffold compared to guided_practice; reduced_scaffold_note explains what was removed and why; learners must produce something

7. `reflection_prompt` (REFLECT) — prompt_text MUST name specific content from THIS session:
   - Prohibited stems: "What did you learn today?", "What went well?", "What were your takeaways?", "What would you do differently?"
   - Required: prompt names a specific concept, decision, case, or moment from this session
   - Example format: "Looking at the [specific scenario/concept from this session] — [specific question about that content]"
   - reflection_type uses exact enum: metacognitive | conceptual | transfer-intent | belief-shift
   - duration_minutes

8. `transfer_connection` (TRANSFER) — application_scenario references the specific transfer_context from project_brief (not generic "apply this to your work"); action_prompt is observable and specific; support_structure names a specific post-session mechanism (not "practice on the job"); duration_minutes

**DCR check (run before generating each session):**
Check: Is skill_type == open? Is this session's primary bloom_level Analyze, Evaluate, or Create?
If BOTH true → DCR fields are REQUIRED (not optional):
- `deconstruct_element` — learners take apart an example
- `compare_element` — learners examine 2+ cases side-by-side
- `rebuild_element` — learners construct/reconstruct using identified principles
- `validate_element` — learners test construction against criteria

DCR novice scaffolding (apply when self_direction_level is Stage 1 or Stage 2):
- compare_element uses close comparisons (cases differing on 1-2 dimensions only)
- Provide explicit structural cues in compare_element description
- Learners do not choose their own comparison cases — pairs are provided
- Note this scaffolding approach explicitly in the compare_element description
