From .claude/reference/schemas/stage-04-modules.md (required fields per module):
```
module_id              — Format: M-1, M-2, M-3
module_name            — Descriptive, not "Module 1"
learning_objectives    — outcome_id refs from Stage 02 (MO-* required)
prerequisite_modules   — module_id refs; valid DAG; [] for first module
content_chunks         — array: chunk_name, duration_minutes, cognitive_load_note
social_learning        — object: all 4 sub-fields required
  activity_type        — enum: individual | paired | small-group | full-cohort
  interdependence_structure — mechanism description (not just a name)
  accountability_mechanism  — individual accountability within group
  group_processing_prompt   — MUST name specific content concept; prohibited: "What went well?" etc.
primary_metaskill      — enum: Exploring | Creating | Innovating | Adapting | Imagining | Communicating
metaskill_activation_activity — specific thinking routine name (not "discussion")
belief_challenging_encounter  — specific belief named; not generic "challenge assumptions"
modality_switches      — array; at least one switch per session in this module
```

Output files (written on approval to workspace/{project-name}/03-modules/):
```
sequence-rationale.md          — module order explanation with dependency table
M-1/module-spec.md             — full spec for module 1
M-2/module-spec.md             — full spec for module 2
(one per module)
```

Duration scaling (from stage-04-modules.md):
```
Short  < 2 hrs  → 1 module; all fields still required
Medium 2-16 hrs → 2-3 modules; sequence-rationale required
Long   > 16 hrs → 8-15 modules; full DAG; belief_challenging_encounter per module
```

Gate summary table format (from CONTEXT.md and RESEARCH.md):
```
| Module | Objectives Covered | Sessions Planned | Primary Metaskill | What This Module Challenges |
Plain-language columns only. No schema field names. module_id never shown as "M-1" in gate table.
Sequence rationale: 1-2 sentences above the table.
```

Constraint enforcement sequence (from RESEARCH.md):
```
Step 1 — DAG validation: no circular dependencies in prerequisite_modules
Step 2 — Learning objectives coverage: every MO-* outcome_id from Stage 02 appears in at least one module
Step 3 — Social learning completeness: all 4 sub-fields present; group_processing_prompt names specific content
Step 4 — Thinking routine specificity: metaskill_activation_activity is a named routine, not generic
Step 5 — Belief encounter specificity: belief_challenging_encounter names the specific belief challenged
Step 6 — Record changes
```

Anti-patterns to avoid (from RESEARCH.md):
- Progressive file writing: generate all, gate, write all simultaneously — never write module files one at a time
- Gate table exposes schema field names (M-1, bloom_level, outcome_id in columns) — translate to plain language
- Generic group_processing_prompts — must check and regenerate if prompt could apply to any module
- ID vocabulary in user-facing output ("module_id", "schema", "enum", "Bloom's")

**Section structure to implement (mirror knz-assessments.md order precisely):**

**Prerequisites (3 checks):**
1. Check workspace exists — read `workspace/*/STATE.md`; if none, respond with init message, stop
2. Check Stage 3 prerequisite — read Stage 3 row from STATE.md; if not `complete`, respond with assessments message, stop
3. Check Stage 4 status:
   - `not-started` → proceed to Generation
   - `in-progress` → if `03-modules/` files exist, re-display gate summary and go to Review Gate; if no files, regenerate from scratch
   - `complete` → respond "Module structure is complete. Run /knz-sessions to generate session content." Stop.

**Persona:**
Expert instructional designer voice. Never say: module_id, bloom_level, outcome_id, schema, enum, prerequisite_modules, social_learning, metaskill, DAG, Bloom's.
Say instead: module, thinking level, learning objectives, prerequisites, collaborative activity, core thinking skill, sequence.

**Generation:**
Load `.claude/reference/schemas/stage-04-modules.md` as generation context. Read:
- `workspace/*/01-outcomes/learning-objectives.md` — all outcome_ids (primary decomposition input)
- `workspace/*/02-assessments/assessment-map.md` — assessment coverage (confirms what each module must set up learners to demonstrate)
- `workspace/*/00-project-brief/project-brief.md` — `contact_hours`, `skill_type`, `self_direction_level`, `modality`, `transfer_context`

Generate the full module structure silently (no running commentary). Apply duration scaling:
- Short (< 2 hrs): 1 module; all fields required
- Medium (2–16 hrs): 2–3 modules; sequence-rationale required
- Long (> 16 hrs): 8–15 modules; full DAG

For each module, generate all required fields: module_id, module_name, learning_objectives (MO-* refs), prerequisite_modules (DAG), content_chunks (with cognitive_load_note matching self_direction_level), social_learning (all 4 sub-fields), primary_metaskill (6-value enum), metaskill_activation_activity (named thinking routine), belief_challenging_encounter (specific belief named), modality_switches.

**Constraint Enforcement (runs before any output shown):**
All corrections silent. Track changes for transparency note.

Step 1 — DAG validation: Verify prerequisite_modules forms no circular dependencies across all modules. If circular dependency found: auto-reorder modules to break the cycle, record the reordering.

Step 2 — Learning objectives coverage: Cross-reference every MO-* outcome_id from `learning-objectives.md` against module learning_objectives arrays. For any MO-* not assigned to a module: add it to the most semantically appropriate module. Record additions.

Step 3 — Social learning completeness: For each module, verify social_learning has all 4 sub-fields with correct types. Check group_processing_prompt against this test: does it name a specific concept from this module's content? Prohibited patterns: "What went well?", "What would you do differently?", "How did your group work together?", "What were your takeaways?". If generic: regenerate with explicit instruction to name the [module topic] concept from the activity. Record regenerations.

Step 4 — Thinking routine specificity: Verify metaskill_activation_activity is a named thinking routine (e.g., "See-Think-Wonder", "Claim-Support-Question", "Connect-Extend-Challenge", "I Used to Think / Now I Think", "Think-Pair-Share" with specific framing). Generic labels like "discussion", "reflection", "group activity" are not acceptable. If generic: regenerate with a named routine. Record changes.

Step 5 — Belief encounter specificity: Verify belief_challenging_encounter names a specific belief being challenged (e.g., "The belief that profit equals success — surfaced through a case where high-revenue businesses fail due to cash flow problems"). Generic "challenge your assumptions" language is not acceptable. If generic: regenerate. Record changes.

Step 6 — Record changes: Track all auto-corrections for transparency note.

**Output Presentation:**
Show validation badge if no corrections: "All modules cover their objectives — sequence and collaborative activities validated."
Show transparency note if corrections made (confident tone): "I adjusted [N] item(s) — [brief example]. Here's the full module structure:"

Then display the gate summary table:

```
## Your Module Structure

[1-2 sentence sequence rationale in plain language — why this order serves learners]

| Module | Objectives Covered | Sessions Planned | Core Thinking Skill | What This Module Challenges |
|--------|--------------------|------------------|---------------------|-----------------------------|
| [module_name] | [N objectives] | [N sessions] | [primary_metaskill plain name] | [paraphrase of belief_challenging_encounter] |
```

Column rules:
- "Module" shows module_name (not module_id like M-1)
- "Objectives Covered" shows count only (not IDs)
- "Sessions Planned" shows count
- "Core Thinking Skill" shows metaskill plain name (Exploring, Creating, etc.)
- "What This Module Challenges" is a one-phrase paraphrase of belief_challenging_encounter — what the module disrupts, in learner language

After table: "Does this structure look right for your program?"

**Review Gate (Module Structure Gate):**
Silently update STATE.md: `Review Gates` → Module-Structure: `pending`

Use AskUserQuestion with three options:
- "Approve and continue" — write files, update STATE.md, show next step
- "I have concerns" — free-text → full regeneration → re-run all 5 constraint steps → re-present gate
- "Start this over" — destructive confirmation gate

**On "Approve and continue":**
1. Write to `workspace/{project-name}/03-modules/`:
   - `sequence-rationale.md` — Overall sequence explanation (1-2 paragraphs grounded in backward design) + Module Dependencies table (Module | Requires | Rationale)
   - `M-1/module-spec.md`, `M-2/module-spec.md`, etc. — Full specification per module with ALL required schema fields (load stage-04-modules.md as context before writing; output must use exact enum values)
   All files written simultaneously, not progressively.
2. Silently update STATE.md:
   - Stage 4 status: `complete`, Completed: {today's date}
   - Review Gates → Module-Structure: `approved`, Approved: {today's date}
   - Session Continuity → Next Action: Run /knz-sessions to generate session content
3. Show: "Module structure approved. Your program has [N] modules covering [X] learning objectives. Next: `/knz-sessions` to generate the session content."

**On "I have concerns":**
Ask "What would you like to change?" Take free-text. Regenerate the FULL module structure using intake + outcomes + assessments + feedback. Re-run all 5 constraint enforcement steps. Re-display transparency note or badge, then gate summary. Re-present gate. Do not update STATE.md gate status during this branch.

**On "Start this over":**
Nested AskUserQuestion: "Are you sure? This will clear all module structure work and start from the beginning." Options: "Yes, start over" / "Keep what we have."
On "Yes, start over": Reset Stage 4 to `not-started` in STATE.md. Reset Module-Structure gate to `not-reached`. Regenerate from scratch with full constraint enforcement.
On "Keep what we have": Return to gate summary, re-present gate.

**State Management Rules:**
All STATE.md reads/writes are silent. Never say: "Updating STATE.md", "Saving progress", "I'm recording that", "Let me check the stage status".
Files written ONLY in the "Approve and continue" branch. 03-modules/ files are never written before approval.
